### 🎯 考试导向原则（v1.6 — 基础保留 + 简洁 + 考试导向）

> **用户的目标是期末考试作出题目，不是深入研究/掌握理论。**
>
> **保留什么**（必须有的）：
> - **基础概念 / 定义**：一句话点到为止（如"既有大小又有方向的量称为向量"），不要砍
> - **关键运算性质 / 定律**：该有的一个都不能少（交换律、分配律、结合律等），不能为简洁而删
>
> **简化什么**（不要写太复杂）：
> - **不要**展开成教科书式的长篇定义 / 严格推导 / 完整证明
> - **不要**堆砌"显然易证"、"由定义可知"之类的填充
> - **点到为止**：能让用户明白意思即可，不必每一步都写
>
> **叠加什么**（在基础之上额外给）：
> - 凡是能用「看到 XX → 做 YY」格式表达的**判定规则 / 解题口诀 / 做题技巧**，**必须**写出来
> - 凡是考试常用的快速方法（判定口诀、记忆口诀、判别式、套用模板），**必须**给
>
> **判断标准**：在考场上看到这个知识点时，能不能立刻动笔？不能，就是笔记没写到位

## 📝 笔记整理规则（v1.0 — 2026-06-29）

> 适用于用户提供的随笔型学习笔记（如 `raw/素材/第X章.md`），生成 `_完整.md` 修订版。

### 触发条件

- 用户说"整理笔记""修订补充""做一份完整版"等
- 源文件包含 `// 开头的提示词`（给 Claude 的指令）
- 源文件存在明显错别字、漏字、语序混乱

### 工作流程

1. **通读原笔记**，标注：
   - 所有错别字（按上下文语义还原）
   - 所有 `//` 开头的提示词（给 Claude 的指令）
   - 缺漏的内容（只有标题/列表项但没正文）
2. **按 `//` 提示词执行规则**：
   - 提示词明确要求"扩展题型/补充例子" → 可以扩展
   - 提示词只要求"形式化准确、符号正确、补充完整" → 只补全已有的内容，不扩张新知识
3. **生成新文件**：
   - 命名格式：`原文件名_完整.md`（如 `1.集合.md` → `1.集合_完整.md`）
   - 放在**与原文件同一目录**（便于对照）
   - 删除所有 `//` 提示词
4. **核心原则**：
   - **🎯 考试导向**（最高优先级）：用户目标是期末考试作出题目，不是深入研究理论。
     - **保留**：基础概念 / 定义（一句话点到为止），关键运算性质 / 定律（该有的一个不能少）
     - **简化**：不展开成教科书式长篇定义 / 严格推导 / 完整证明，不堆砌"显然易证"之类填充
     - **叠加**：凡是能用「看到 XX → 做 YY」表达的判定规则 / 解题口诀 / 做题技巧，必须写出来；考试常用的快速方法（判定口诀、记忆口诀、判别式、套用模板）必须给
   - **跟随用户思路**：保留原文的结构、章节顺序、个人化表述
   - **不扩张内容**：除非 `//` 提示词明确允许，否则不引入新概念、新题型
   - **形式化优先**：符号、公式、定理名称必须准确
   - **错别字修订**：根据上下文推断原意，不臆造
   - **纯笔记输出**：生成的 `_完整.md` 只包含笔记本身（标题 + 正文），**不写任何顶部/底部的修订说明、生成元注释、操作记录**。修订过程不污染笔记，过程信息只出现在对话里

### 默认假设

- 当用户说"不要扩张内容"时，严格遵循——只补全已写出的空位
- 当 `//` 提示词允许扩展时（如"适度扩展题型"），可以加少量例题，但不应改变原结构

### 输出检查清单

- [ ] 所有 `//` 提示词已删除
- [ ] 错别字已修订（语义通顺）
- [ ] 缺漏内容已补全（仅限原结构内的）
- [ ] 数学符号准确（$\in$、$\subseteq$、$\cup$、$\cap$ 等）
- [ ] 章节顺序与原笔记一致
- [ ] 未引入原笔记没有的新概念
- [ ] 笔记顶部/底部无任何修订说明、生成元注释、操作记录等非笔记内容
- [ ] **🎯 考试导向**：
  - 关键判定规则都用「看到 XX → 做 YY」/口诀/公式给出
  - 基础概念 / 定义 保留（一句话点到为止即可）
  - 关键运算性质 / 定律 没被砍掉
  - 没有教科书式长篇定义 / 严格推导 / 完整证明占据正文位置

## 🗂️ Wiki Schema 规则（**当前生效**）

> 当前已切换到 wiki 模式。下面的规则在每次 Claude 会话中生效。

### Wiki 维护工作流

#### 1. Ingest（摄取新原始资料）

当用户在 `raw/素材/` 写入新的原始笔记（如 `1.常微分方程.md`）后：

1. **通读原始笔记** + 已有的 `_完整.md`（如有）
2. **创建来源摘要页**：`wiki/来源/YYYY-MM-DD-<章节名>.md`
   - frontmatter: `type: source, sources: [原始文件名], created: 今日, updated: 今日`
   - 内容：章节概要、关键公式清单、与其他章节的关联
3. **创建/更新概念页**：`wiki/概念/<概念名>.md`
   - 每章 4-5 个核心概念（大粒度）
   - frontmatter: `type: concept, sources: [原始文件名], tags: [...], created: 今日, updated: 今日`
   - 内容：定义、关键定理、典型例题链接、做题套路、与其他概念关联
4. **更新 `wiki/index.md`**：列出新增的页面
5. **追加 `wiki/log.md`**：`## [YYYY-MM-DD] ingest | 章节名`

#### 2. Query（查询）

当用户提问"什么是 X""X 和 Y 的区别"等：

1. 读 `wiki/index.md` 找相关页面
2. 深入读相关 `wiki/概念/` 页
3. 综合回答；如果答案里有"新比较 / 新关联"且有价值，归档为新的对比页或更新现有概念页

#### 3. Lint（健康检查）

定期（用户问"检查一下"时）：

1. 查孤儿页面（无入链）
2. 查过时信息（来源页比概念页新）
3. 查缺失的交叉引用
4. 查矛盾（同一概念在不同页说法不一致）

### Wiki 页面约定

所有 wiki 页面都应有 YAML frontmatter：

```yaml
---
type: source | concept | comparison
tags: [tag1, tag2]
sources: [来源文件名]
created: YYYY-MM-DD
updated: YYYY-MM-DD
---
```

### 创建时机规则

| 页面类型 | 创建条件 |
|---------|---------|
| **来源摘要页** | **每篇原始资料都对应一个**（始终创建） |
| **概念页** | 一个章节对应 4-5 个核心概念（大粒度） |
| **对比页** | 跨章节的比较（如第一型 vs 第二型曲线积分）|

### 索引与日志格式

- `wiki/index.md` 格式：表格列出所有页面，按类型分类
- `wiki/log.md` 格式：`## [YYYY-MM-DD] 操作类型 | 标题`（**追加式**，可用 grep 解析）
